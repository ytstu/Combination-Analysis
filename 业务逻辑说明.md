# 组合装拆解分析程序业务逻辑说明

## 1. 项目定位

这个程序是一个可直接运行的脚本，用来把默认输入 Excel 中的 `原始商品编码` 解析成组合装数据，并整理成可导出的组合商品清单。

它当前的核心用途是：

- 从输入表中识别可能的组合装编码
- 拆出基础商品编码和数量
- 去默认商品资料库中补齐商品信息
- 生成组合商品名称
- 输出为一张组合资料表

当前绝大多数业务逻辑都集中在 `main.py` 中。

## 2. 运行方式

程序入口在 `main.py`。

当前运行命令：

```bash
python3 main.py
```

当前版本固定使用默认输入文件，不再支持手动传入输入文件，也不支持手动指定商品资料库、组合资料库和输出路径。

## 3. 默认文件路径

程序固定使用以下默认路径：

- 输入文件：`data/input/模拟-i输入-讲解.xlsx`
- 商品资料库：`data/input/商品资料.xlsx`
- 组合资料库：`data/input/组合资料.xlsx`
- 输出目录：`data/output`

输出文件名格式为：

```text
组合装数据MMDD.xlsx
```

例如在 3 月 4 日运行时，输出文件名会是：

```text
组合装数据0304.xlsx
```

## 4. 输入和输出

### 输入要求

默认输入 Excel 必须包含字段：

- `原始商品编码`

如果没有这个字段，程序会在后续处理时因缺少该列而报错。

### 输出字段

最终导出的 Excel 只包含以下 4 列：

- `组合商品编码`
- `组合商品名称`
- `商品编码`
- `数量`

这与当前组合资料库的结构一致。

## 5. 核心业务流程

程序的核心处理逻辑在 `ExcelDataService.process_data()` 中，处理流程如下。

### 5.1 读取默认输入表

程序读取固定默认输入文件 `data/input/模拟-i输入-讲解.xlsx`。

### 5.2 去重

按 `原始商品编码` 去重，同一个编码只保留一条。

### 5.3 过滤非组合装编码

程序只保留包含 `*` 的编码。

这是当前代码中判断“像组合装编码”的唯一规则。比如：

- `MJ12-2*2` 会保留
- `NOASTERISK` 会被过滤

### 5.4 判断是否已存在

程序会对每条 `原始商品编码` 做两次检查：

- 是否已存在于商品资料库的 `商品编码`
- 是否已存在于组合资料库的 `组合商品编码`

如果任意一边已存在，这条记录就不会继续处理。

### 5.5 拆分编码

程序按最后一个 `*` 来拆分编码：

- `*` 前面的部分记为 `倍数前`
- `*` 后面的部分记为 `倍数后`

例如：

```text
MJ12-2*2 -> 倍数前 = MJ12-2, 倍数后 = 2
```

### 5.6 查询基础商品信息

程序用 `倍数前` 去商品资料库中按 `商品编码` 查找对应商品，并准备读取这些字段：

- `商品名称`
- `尺寸规格(mm)`
- `数量(pcs)`
- `颜色`

### 5.7 生成组合商品名称

程序当前使用下面的规则拼接组合商品名称：

```text
商品名称 + 尺寸规格(mm) + (数量(pcs) * 倍数后) + 颜色
```

也就是说：

- 先取基础商品的原始名称
- 再拼接尺寸
- 再把基础商品的 `数量(pcs)` 乘以倍数，得到组合后的总数量
- 最后拼接颜色

示例：

如果基础商品是：

- 商品编码：`MJ12-2`
- 商品名称：`50个随机混色小乌龟8mm`
- 数量(pcs)：`50`
- 尺寸规格(mm)：`8`
- 颜色：`混色`

输入编码为：

```text
MJ12-2*2
```

则程序生成的组合商品名称为：

```text
50个随机混色小乌龟8mm8100混色
```

### 5.8 导出结果

最终程序会把处理结果导出为 Excel，写入 `data/output` 目录下。

## 6. 当前代码中的业务假设

从当前实现来看，程序默认采用以下业务假设：

- 输入中带 `*` 的编码，代表“可能的组合装编码”
- `*` 前面是基础商品编码
- `*` 后面是数量或倍数
- 如果这个编码还没有在商品库或组合库中出现，就把它作为新的组合装候选项导出

因此，这个程序更像是“组合装候选数据整理工具”，而不是完整的商品管理系统。

## 7. 当前实现中的注意点

下面这些点需要团队明确，因为它们会直接影响结果是否符合业务预期。

### 7.1 单品存在判断检查的是完整原始编码

当前代码检查 `单品是否存在` 时，用的是完整的 `原始商品编码`，例如 `MJ12-2*2`。

这意味着它不是在判断基础商品 `MJ12-2` 是否存在，而是在判断带 `*` 的整串编码是否直接出现在商品资料库里。

如果业务目标是“检查基础单品是否存在”，那当前逻辑并不严格匹配这个目标。

### 7.2 基础商品查不到时，记录仍可能导出

即使 `倍数前` 在商品资料库中找不到，程序也不会自动过滤这条记录。

这种情况下：

- `商品编码` 和 `数量` 仍会出现在导出结果中
- `组合商品名称` 可能为空

### 7.3 存在未使用的中间字段

处理中生成了若干中间列，例如：

- `临时名称`
- `提取后`

其中 `提取后` 当前没有参与最终结果，属于未实际使用的中间字段。

### 7.4 输出文件可能被覆盖

输出文件名只按日期生成，不包含输入文件名或时间戳。

因此同一天多次运行时，后一次导出会覆盖前一次结果。

## 8. 代码结构说明

当前 `main.py` 中的主要职责划分如下：

- `ExcelDataService`
  - 负责读取商品资料库和组合资料库
  - 负责读取输入文件
  - 负责执行核心数据处理
  - 负责构造最终导出表
- `resolve_output_path()`
  - 负责生成默认输出文件路径
- `run()`
  - 负责串联完整流程：读取默认输入文件、加载数据库、处理数据、导出结果
- `main()`
  - 程序入口，直接调用 `run()`

## 9. 一句话总结

这个程序的业务本质是：

把输入表中的“疑似组合装编码”拆解成基础商品和数量，再结合商品资料库生成组合商品信息，并导出为组合资料表。
